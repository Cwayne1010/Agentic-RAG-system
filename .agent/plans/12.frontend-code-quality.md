## Task 1 — Fix `state_referenced_locally` warnings (2 files)

### Background

Svelte 5 warns when a `$state` or `$props` value is read **outside** of a reactive context (`$derived`,
`$effect`, template). The value is captured once at initialisation — changes after that are invisible.

**Warning locations:**

- [ToolCallDisplay.svelte:19](../../../frontend/src/lib/components/chat/ToolCallDisplay.svelte#L19) — `stateKey` read directly in `$state(...)` initialiser
- [documents/+page.svelte:17](../../../frontend/src/routes/documents/+page.svelte#L17) — `data.documents` read directly in `$state(...)` initialiser

**Root cause in `ToolCallDisplay.svelte`:**

```svelte
// ❌ stateKey is a prop — read once, never updates if parent changes it
let open = $state(stateKey ? localStorage.getItem(`tcd:${stateKey}`) === '1' : false);
```

`open`'s *initial value* depends on `stateKey`, but is fine as one-time init — the real fix is to
silence the warning by moving the prop read into a function call that makes intent explicit:

```svelte
// ✅ wrap in a function so Svelte knows this is intentional one-time init
function initialOpen(): boolean {
    return stateKey ? localStorage.getItem(`tcd:${stateKey}`) === '1' : false;
}
let open = $state(initialOpen());
```

**Root cause in `documents/+page.svelte`:**

```svelte
// ❌ data is a $props value — captured once
let documents = $state<Document[]>(data.documents);
```

`data.documents` is used to seed local state (intentional one-time init). Same pattern — extract to
a function:

```svelte
// ✅ make the one-time init intent explicit
let documents = $state<Document[]>(data.documents ?? []);
```

Actually for the page, the idiomatic Svelte 5 fix is to use `$derived` if documents should stay in
sync with the load function, or keep `$state` with an explicit seed if it's intentionally local
(editable). Since this page adds/removes documents locally after upload, `$state` seeded from `data`
is correct — just suppress with a no-reactive-tracking read:

```svelte
// Extract to untrack — tells Svelte this read is intentionally non-reactive
import { untrack } from 'svelte';
let documents = $state<Document[]>(untrack(() => data.documents));
```

### Steps

1. In [ToolCallDisplay.svelte](../../../frontend/src/lib/components/chat/ToolCallDisplay.svelte):
   - Add `function initialOpen()` before the `let open` line
   - Change `let open = $state(stateKey ? ... : false)` to `let open = $state(initialOpen())`

2. In [documents/+page.svelte](../../../frontend/src/routes/documents/+page.svelte):
   - Add `import { untrack } from 'svelte'` to the imports
   - Change `let documents = $state<Document[]>(data.documents)` to use `untrack`

### Validation

```bash
cd frontend && npx svelte-check --threshold warning 2>&1 | grep "state_referenced_locally"
```

**Pass:** No `state_referenced_locally` lines in output
**Fail:** Warning lines still present

## Task 2 — Fix `SettingsModal` accessibility warnings (1 file)

### Background

[SettingsModal.svelte](../../../frontend/src/lib/components/chat/SettingsModal.svelte) has 10 warnings
across two categories:

**Category A — Backdrop `<div role="dialog">` (lines 137–141)**

Two warnings:
1. `a11y_interactive_supports_focus` — elements with interactive roles need `tabindex`
2. `a11y_click_events_have_key_events` — click handlers need a keyboard equivalent

The backdrop div handles clicks to close the modal. The fix is to replace it with a semantically
correct `<dialog>`-like pattern. Since the *inner panel* is the real dialog, the backdrop should be
a plain `<div>` (no role) with both `onclick` and `onkeydown`:

```svelte
<!-- ❌ Current -->
<div
    class="fixed inset-0 ..."
    role="dialog"
    aria-modal="true"
    onclick={handleBackdrop}
>

<!-- ✅ Fixed — role and aria go on the inner panel, backdrop handles keyboard too -->
<div
    class="fixed inset-0 ..."
    onclick={handleBackdrop}
    onkeydown={(e) => e.key === 'Escape' && onclose()}
    role="presentation"
    tabindex="-1"
>
    <div
        role="dialog"
        aria-modal="true"
        aria-labelledby="settings-title"
        class="bg-background relative w-full max-w-lg rounded-xl border shadow-xl"
    >
```

Also add `id="settings-title"` to the `<h2>Settings</h2>` heading so `aria-labelledby` resolves.

**Category B — 8 `<label>` elements without `for` attribute (lines 182–239)**

Each `<label>` wraps an `<Input>` component. Because `Input` is a shadcn component, using `for`
requires knowing the rendered `id`. The correct fix is to either:

- Add matching `id` props to each `Input` and `for` to each `label`, **or**
- Wrap each pair in a `<label>` that *contains* the input (implicit association — no `for` needed)

The current structure already wraps `<label>` and `<Input>` in a `<div>` but the label doesn't
contain the input. The simplest fix with minimal restructuring: add `for`/`id` pairs.

Fields and their `id` values to add:

| Label text | id to use |
|---|---|
| Model (LLM) | `llm-model` |
| Map Model | `map-model` |
| Base URL (LLM) | `llm-base-url` |
| API Key (LLM) | `llm-api-key` |
| Model (Embedding) | `embedding-model` |
| Base URL (Embedding) | `embedding-base-url` |
| API Key (Embedding) | `embedding-api-key` |
| Dimensions | `embedding-dimensions` |

Pattern:
```svelte
<!-- ❌ Current -->
<label class="mb-1 block text-xs text-muted-foreground">Model</label>
<Input bind:value={chatModel} ... />

<!-- ✅ Fixed -->
<label for="llm-model" class="mb-1 block text-xs text-muted-foreground">Model</label>
<Input id="llm-model" bind:value={chatModel} ... />
```

Note: shadcn `Input` accepts and forwards `id` via `$$restProps` / Svelte 5 spread — verify it
renders the `id` on the underlying `<input>` element after fixing.

### Steps

1. Fix backdrop div: move `role="dialog"` + `aria-modal` to the inner panel div, add
   `role="presentation" tabindex="-1"` to outer div, add `onkeydown` Escape handler
2. Add `id="settings-title"` to `<h2>` and `aria-labelledby="settings-title"` to the dialog panel
3. Add `for`/`id` pairs to all 8 label/input pairs in the Models tab
