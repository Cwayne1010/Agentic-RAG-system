# Plan 4: Speculative Pre-Fetched Retrieval

✅ **Simple** — Two files modified. Immediately measurable improvement.

## Problem

Current chat flow makes 2 sequential LLM calls per user message:
1. LLM call 1 (with tools) → decides to call `retrieve_documents`
2. Embedding + pgvector search
3. LLM call 2 (with context) → generates response

TTFT is dominated by LLM call 1 sitting idle before retrieval starts.

## Solution

Pre-fetch retrieval using the user message directly, before any LLM call → 1 LLM call total.

```
Before: LLM (decide) → embed → search → LLM (respond)    [2 LLM calls, sequential]
After:  embed → search → LLM (respond with context)       [1 LLM call]
```

## Files Modified

### `backend/app/routers/chat.py`

Replaced the tool-call loop in `event_stream()` with:
1. `retrieval_service.search(body.message, user_id)` — runs before any LLM call
2. SSE `retrieval` event emitted immediately (chunk_count: 0 if no docs matched)
3. Context injected into system message via `<retrieved_context>` XML tags
4. Single `traced_stream_chat(..., use_tools=False)` call

### `backend/app/services/openrouter_service.py`

Updated `SYSTEM_PROMPT` — removed tool-calling instructions, added context-tag instructions:
```python
SYSTEM_PROMPT = (
    "You are a helpful AI assistant. When relevant document context is provided "
    "in <retrieved_context> tags, use it to answer the user's question and cite "
    "the source. If no context is provided, answer from your general knowledge."
)
```

`RETRIEVAL_TOOL` kept in file (not deleted) with comment — will be reused in Module 7.

## Trade-off

Always calls `embed_text()` + pgvector search even for non-document questions.
- Embedding cost: ~$0.00002/call
- Embedding latency: ~100-200ms
- Far cheaper than the second LLM call it eliminates (~$0.001+ and 500ms+)
- If chunks=0, context_block="" and LLM answers from general knowledge
