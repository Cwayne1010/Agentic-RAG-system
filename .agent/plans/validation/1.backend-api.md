### D3 — Chunks stored with 1536-dim embeddings (SQL)
```sql
SELECT d.filename,
       COUNT(dc.id) AS chunk_count,
       vector_dims(dc.embedding) AS embedding_dims
FROM documents d
JOIN document_chunks dc ON dc.document_id = d.id
WHERE d.filename = 'test.txt'
GROUP BY d.filename, vector_dims(dc.embedding);
```
**Pass:** Row with `chunk_count >= 1` and `embedding_dims = 1536`
**Fail:** No rows, or embedding_dims ≠ 1536

### D4 — Delete document cascades chunks (SQL)
```sql

WHERE document_id NOT IN (SELECT id FROM documents);
```
**Pass:** 0 orphaned chunks
**Fail:** Orphaned chunks remain

### RM3 — Duplicate upload rejected with 409 (curl)
```bash
printf "hello dedup world" > /tmp/test_dedup.txt
# First upload — should succeed:
curl -s -w "\nHTTP %{http_code}" -X POST http://localhost:8000/api/documents/upload \
  -H "Authorization: Bearer $TOKEN" -F "file=@/tmp/test_dedup.txt"
# Wait for status=completed in DB or UI, then upload again:
curl -s -w "\nHTTP %{http_code}" -X POST http://localhost:8000/api/documents/upload \
  -H "Authorization: Bearer $TOKEN" -F "file=@/tmp/test_dedup.txt"
```
**Pass:** Second upload returns `HTTP 409`; body contains `"Duplicate"` and the original filename
**Fail:** Second upload returns 200 and creates a second document record

### RM4 — Incremental update: same filename, new content (curl)
```bash
printf "version one content" > /tmp/update_test.txt
curl -s -X POST http://localhost:8000/api/documents/upload \
  -H "Authorization: Bearer $TOKEN" -F "file=@/tmp/update_test.txt"
# Wait for completed, note the returned id as OLD_ID. Then:
printf "version two - completely different content" > /tmp/update_test.txt
curl -s -w "\nHTTP %{http_code}" -X POST http://localhost:8000/api/documents/upload \
  -H "Authorization: Bearer $TOKEN" -F "file=@/tmp/update_test.txt"
```
**Pass:** Second upload returns `HTTP 200` with a new `id`; verify only one row in DB:
```sql
SELECT id, filename, status FROM documents WHERE filename = 'update_test.txt';
```
Pass: 1 row only. Fail: 2 rows with same filename.

### RM5 — Re-upload of failed document is allowed (manual)
Manually set a document to failed, then re-upload the same file:
```sql

UPDATE documents SET status = 'failed', error_message = 'test failure' WHERE id = '<doc_id>';
```
Then upload the same file via the UI or curl.

**Pass:** Upload returns `HTTP 200`; new ingestion begins
**Fail:** `HTTP 409` — failed docs should never block re-upload

### RM6 — Cross-user: same content uploadable by both users (curl)
Using Clocker's token, upload a file that Flocker already has as completed:
```bash
curl -s -w "\nHTTP %{http_code}" -X POST http://localhost:8000/api/documents/upload \
```

### RM7 — content_hash stored correctly (SQL + Python)
After uploading `test_dedup.txt` from RM3:
```sql
SELECT filename, content_hash, length(content_hash) AS len
FROM documents WHERE filename = 'test_dedup.txt' LIMIT 1;
```
Compute expected hash independently:
```python
import hashlib
print(hashlib.sha256(open('/tmp/test_dedup.txt','rb').read()).hexdigest())
```
**Pass:** `len = 64`; DB value matches Python output exactly
**Fail:** Length ≠ 64, or hash mismatch
