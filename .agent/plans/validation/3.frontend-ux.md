# Validation — Part 3: UX / Frontend

All tests require a running browser session at `http://localhost:5173`.

---

## Pre-Flight

### P2 — Frontend running
Open `http://localhost:5173`
**Pass:** Auth form (login/signup) displayed
**Fail:** Blank page, error, or redirect

---

## Auth

### A1 — Login with valid credentials
1. Open `http://localhost:5173`
2. Enter `flocker@login.com` / `flocker` → Click Login

**Pass:** Chat UI loads with sidebar; no auth form visible
**Fail:** Error message or auth form persists

### A2 — Auth gating blocks unauthenticated access
1. Open a fresh private/incognito window
2. Navigate to `http://localhost:5173`

**Pass:** Auth form shown immediately; no chat UI
**Fail:** Chat UI accessible without login

### A4 — Logout clears session
1. Login as `flocker@login.com`
2. Click Logout in the sidebar

**Pass:** Auth form shown again; chat UI gone
**Fail:** Chat UI persists after logout

---

## Chat

### CH1 — Send a message and receive streaming response
1. Login → Click "New Chat"
2. Type `What is 2 + 2?` → Press Enter

**Pass:** Response streams token-by-token; final answer contains "4"; both messages appear; conversation updates in sidebar
**Fail:** No response, all-at-once response, or sidebar doesn't update

### CH2 — Stop mid-stream
1. Send: `Tell me a very long story about a dragon`
2. Click Stop while streaming

**Pass:** Streaming halts immediately; partial response preserved
**Fail:** Stream continues, or UI resets/crashes

### CH3 — Messages persist on page reload
1. Send 2+ messages in a conversation
2. Hard-reload (Cmd+Shift+R)
3. Select the same conversation from sidebar

**Pass:** All previous messages visible in correct order
**Fail:** Messages lost on reload

**Fail:** No traces, or project is empty

## Documents

### D1 — Upload a text file
1. Create `test.txt` with content: `The capital of France is Paris.`
2. Navigate to `http://localhost:5173/documents`
3. Drag and drop `test.txt` onto the upload zone

**Pass:** Document appears with status `pending` → `processing` → `completed` automatically; chunk count > 0
**Fail:** Upload fails, status stuck, or requires refresh

### D4 (browser) — Delete document removes from list
Click delete next to `test.txt`; verify it disappears from the list immediately.

**Pass:** Document removed from UI without page refresh
**Fail:** Still visible, or requires refresh

### D5 — Upload markdown file
Upload any `.md` file

**Pass:** Accepted and ingested (same flow as D1)
**Fail:** Rejected or silently fails

### D6 — Reject unsupported file type
Attempt to upload a `.jpg` or `.pdf` file

**Pass:** Upload rejected with an error message
**Fail:** File silently accepted or crashes

## Module 3: Record Manager

### RM8 — Duplicate toast shown in UI (browser)
1. Upload any `.txt` file via the Documents page
2. Wait for status to reach `completed`
3. Upload the exact same file again

**Pass:** An amber/yellow warning toast appears with message starting `"Duplicate: this content already exists as..."` — not a red error toast
**Fail:** Generic red error toast, upload appears to succeed, or no feedback at all

### RM9 — Update toast and stale card replaced (browser)
1. Upload a `.txt` file, wait for `completed`
2. Edit the file locally (change its content), then upload the same filename again

**Pass:** A green success toast says `"Updated <filename>"`; the old document card is replaced by the new one (not duplicated); new card cycles through `pending → processing → completed`
**Fail:** Old card remains alongside new one, duplicate error shown, or update not reflected

## Module 4: Metadata Extraction

### M4-UX1 — Doc type badge and topic chips visible after indexing
1. Login as `flocker@login.com`
2. Navigate to Documents page
3. Upload a `.txt` file and wait for status to reach `Indexed`

**Pass:** Document card shows a grey doc type badge (e.g. `report`, `article`) and up to 3 topic chips below the file info row
**Fail:** No badges/chips visible after indexing, or badges visible while still `pending`/`processing`

### M4-UX2 — Retrieval info toast shown in chat
1. Login → open Chat
2. Send a message about a topic covered in an uploaded document (e.g. "summarise the document")

**Pass:** A brief info toast appears showing retrieved chunk count and source filename(s), e.g. "Retrieved 3 chunks from report.txt"
**Fail:** No toast shown, or toast appears even when chunk_count is 0

### M5-UX1 — PDF drag-and-drop accepted and indexed

Steps:
1. Go to http://localhost:5173/documents
2. Drag a PDF file onto the upload zone
3. File appears in list with "Indexing…" status
4. Wait; status changes to "Indexed" with chunk count shown

**Pass:** PDF processes end-to-end in the UI
**Fail:** Drop zone ignores file, or document stays in failed/pending state

### M5-UX2 — Unsupported file type silently ignored

Steps:
1. Drag a `.xlsx` or `.jpg` file onto the upload zone

**Pass:** Nothing happens (file silently filtered before upload)
**Fail:** Upload attempted; error toast appears; or backend 400 shows

### CP1 — Chunk progress shown during processing

Steps:
1. Login → go to `/documents`
2. Upload a large document (>500KB PDF or multi-page DOCX)
3. Watch the document row while status is "Processing"

**Pass:** Sub-line reads "Embedding X / N chunks" and the number increments in real-time until complete; once status changes to "Indexed" the line switches to "N chunks"
**Fail:** No progress text shown; counter stays at 0; or text remains after indexing

### CP2 — Chunk progress absent for small/fast documents

Steps:
1. Upload a small `.txt` file (a few lines)
2. Watch the document row

**Pass:** Document moves through parsing → processing → indexed quickly; "Embedding" text may flash briefly or not appear at all; final state shows chunk count
**Fail:** Progress text stuck at "Embedding 0 / N chunks" after indexing completes

### CP3 — Parsing badge appears immediately on upload

Steps:
1. Login → go to `/documents`
2. Upload a large PDF (>500KB)
3. Watch for the document row to appear

**Pass:** Document row appears immediately (within ~1s of clicking upload) with a purple "Parsing" badge and "Reading document…" sub-line; upload zone is no longer blocked
**Fail:** Document row does not appear until parsing completes (many seconds of nothing); or badge shows "Pending" or "Processing" instead of "Parsing"

## Module 8: Sub-Agents

### M8-UX1 — Full-context toggle is gone from the MessageInput UI
1. Login → chat interface
2. Inspect the message input area
**Pass:** No full-context toggle button or "Full context mode active" indicator visible
**Fail:** Toggle or indicator still present

### M8-UX2 — "Document agent" parent pill + child pills appear for summarise request
1. Ask "Summarise my Tickler document"
2. Watch for tool call pills
**Pass:** "Document agent" pill appears; child pill "Sub-agent: searching docs" appears below it
**Fail:** No nested pills or wrong labels

### M8-UX3 — "Summarise all documents" triggers all-docs path with scanning progress visible
1. Ask "Summarise all my documents"
2. Watch for scanning/map-reduce events
**Pass:** "Document agent" pill shows → response includes scan summary (X docs, Y relevant)
**Fail:** No scanning indication or normal retrieval path taken

### M8-UX4 — Existing tool pills (query_database, web_search) still work flat/unaffected

2. Ask "What is the latest news?" → should trigger web_search

**Fail:** Pills missing, nested incorrectly, or tools not triggered
